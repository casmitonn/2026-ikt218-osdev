# Assignment Three — Interrupt Handling

## Concept of Interrupts in Operating Systems

A CPU without interrupts must constantly poll every device to check whether something needs attention — reading the keyboard port in a loop, checking whether a disk transfer finished, etc. This wastes cycles and scales poorly as the number of devices grows.

An **interrupt** is a signal to the CPU that an event requires immediate attention. When an interrupt fires:

1. The CPU finishes the current instruction.
2. It saves its state (EIP, CS, EFLAGS — and ESP/SS on privilege change) onto the stack.
3. It looks up the handler address in the IDT using the interrupt vector number.
4. It jumps to that handler.
5. The handler runs, then executes `iret`, which restores saved state and resumes normal execution.

This means the CPU can sit idle (with `hlt`) and be woken only when work actually arrives — no polling required.

Interrupts come in two kinds:

- **Hardware interrupts (IRQs):** generated by physical devices (keyboard, timer, disk). Routed through the PIC to the CPU.
- **Software interrupts (exceptions and traps):** generated by the CPU itself (divide by zero, page fault) or deliberately by software (`int N`).

Interrupts are the foundation of multitasking: a timer IRQ fires periodically, the OS saves the current task's state and restores another's — giving the illusion of simultaneous execution.

## Stage 1 — Interrupt Descriptor Table

### What

The IDT is a hardware table that tells the CPU where to jump when an interrupt or exception occurs. It holds up to 256 gate descriptors, one per vector. We implement vectors 0–31 (CPU exceptions) here, with vectors 0x20–0x2F added in Stage 3 for hardware IRQs.

### Why

Without an IDT loaded, any interrupt or exception causes a triple fault and silently reboots the machine. The IDT must be installed before enabling hardware interrupts.

### How

**Gate descriptor layout (8 bytes, `struct idt_entry`):**

```
Bits 63:48  offset_high   — upper 16 bits of the handler address
Bits 47:40  type_attr     — gate type + privilege flags
Bits 39:32  zero          — always 0 (reserved by Intel)
Bits 31:16  selector      — code segment selector (0x08, kernel CS)
Bits 15:0   offset_low    — lower 16 bits of the handler address
```

`type_attr = 0x8E` decodes as:
- Bit 7 (P)      = 1 → gate is present
- Bits 6:5 (DPL) = 00 → ring 0 only
- Bit 4          = 0 → not a storage segment
- Bits 3:0       = 1110 → 32-bit interrupt gate

A 32-bit interrupt gate automatically clears the IF flag on entry, preventing nested faults while the handler runs.

**IDT pointer (`struct idt_ptr`):**

```
Bits 47:16  base   — linear address of the IDT array
Bits 15:0   limit  — byte size of the IDT minus 1
```

This 6-byte structure is passed directly to the `lidt` instruction in `idt_load`.

**ISR stubs (`src/idt.asm`):**

CPU exceptions split into two groups depending on whether the CPU pushes an error code automatically:

- **With error code:** vectors 8, 10, 11, 12, 13, 14, 17
- **Without error code:** all others

To keep a uniform stack layout for the C handler, stubs without an error code push a dummy `0` first. Both groups then push their vector number and jump to `common_isr_stub`.

`common_isr_stub` saves all registers (`pusha`), saves and reloads the data segment to the kernel selector (0x10), passes a pointer to the full register frame to `isr_handler` in C, then restores everything and returns with `iret`.

**Stack frame passed to `isr_handler` (`registers_t`):**

```
Low address (top of stack)
  ds
  edi, esi, ebp, esp, ebx, edx, ecx, eax   ← saved by pusha
  int_no, err_code                           ← pushed by stub
  eip, cs, eflags                            ← pushed by CPU
High address
```

## Stage 2 — Interrupt Service Routines (ISRs)

### What

An ISR (Interrupt Service Routine) is the function the CPU calls when an exception fires. `isr_handler` in `src/idt.c` is the single C handler reached by all 32 exception stubs. It identifies the exception by vector number and prints its name.

### Why

Without ISRs, any CPU exception silently triple-faults and reboots. With named output we can see exactly what went wrong. Halting on #DF, #GP, and #PF is correct because returning from these in a bare kernel leads to undefined behaviour.

### How

A 32-entry static string table maps vector numbers to their Intel-defined names:

```c
static const char *exception_names[32] = {
    "Division Error", "Debug", "Non-Maskable Interrupt", ...
};
```

The handler prints `Exception N: <name>`, appends the error code if non-zero, then halts on vectors 8, 13, and 14.

**Testing:** three software interrupts are fired from `kernel.c`:

```c
asm volatile("int $0x0");  // Division Error
asm volatile("int $0x3");  // Breakpoint
asm volatile("int $0x6");  // Invalid Opcode
```

The CPU treats these as software-triggered exceptions — it saves state, calls the handler, and `iret`s back to the next instruction. All three print their names and return, confirming the full stub → handler → iret path works before hardware interrupts are enabled.

## Stage 3 — Interrupt Requests (IRQs)

### What

An IRQ (Interrupt Request) is a hardware interrupt raised by a peripheral device. The 8259A Programmable Interrupt Controller (PIC) collects IRQ signals from up to 15 devices and forwards them to the CPU as interrupt vectors.

There are two cascaded chips:
- **PIC1 (master):** handles IRQ0–7
- **PIC2 (slave):** handles IRQ8–15

### Why — Remapping

After reset, PIC1 fires on vectors 0x08–0x0F and PIC2 on 0x70–0x77. Vectors 0x08–0x0F overlap directly with the CPU exception vectors (#DF, #TS, #NP, #SS, #GP, #PF, #MF). A hardware IRQ delivered at the wrong moment would be mistaken for a CPU exception and crash the kernel.

We remap:
- PIC1 → vectors 0x20–0x27
- PIC2 → vectors 0x28–0x2F

This places all hardware IRQs above the 32 CPU exception vectors, eliminating the conflict.

### How — PIC Initialisation (`src/pic.c`)

Remapping requires sending four Initialisation Command Words (ICWs) to each PIC in order:

| ICW | Value | Meaning |
|-----|-------|---------|
| ICW1 | 0x11 | Begin init sequence; ICW4 will follow |
| ICW2 | 0x20 / 0x28 | New vector base offset |
| ICW3 | 0x04 / 0x02 | PIC1: slave on IRQ2 (bit mask); PIC2: cascade ID |
| ICW4 | 0x01 | 8086 mode |

`io_wait()` is called between each write. It writes to port 0x80 (the POST diagnostic port), giving hardware time to process each command.

After init, all IRQ lines are masked (0xFF to both data ports). Individual lines are unmasked only when a handler is registered, preventing spurious interrupts.

**`pic_send_eoi(irq)`** signals end-of-interrupt to the PIC(s). If the IRQ came from PIC2 (irq ≥ 8), EOI is sent to PIC2 first, then PIC1. Without EOI, the PIC holds the IRQ line and no further interrupts on that line fire.

**`pic_unmask_irq(irq)`** reads the current mask register, clears the bit for the requested line, and writes it back — enabling one IRQ without disturbing others.

### How — IRQ Stubs and Dispatch (`src/idt.asm`, `src/idt.c`)

Sixteen IRQ stubs (irq0–irq15) follow the same pattern as ISR stubs but always push dummy 0 for the error code (hardware never supplies one), then jump to `common_irq_stub`.

`irq_handler` in `src/idt.c`:
1. Computes the IRQ number: `int_no - 0x20`
2. Calls the registered function from `irq_handlers[16]` if one exists
3. Always calls `pic_send_eoi` — the PIC must be released even if no handler is registered

`irq_install_handler(irq, fn)` stores a function pointer into the dispatch table. The keyboard (IRQ1) and future PIT (IRQ0) use this to plug in without touching the IRQ infrastructure.

## Stage 4 — Keyboard Logger (IRQ1)

### What

The PS/2 keyboard is wired to IRQ1. On each keypress or release the keyboard controller sends a scancode byte to port 0x60 and fires IRQ1.

### Why

Keyboard input is needed for the music player to control playback. It also satisfies Task 4 of the assignment directly.

### How — Scancodes

The PS/2 keyboard sends set-1 scancodes by default:
- **Make code** (key pressed): 0x00–0x7F
- **Break code** (key released): make code | 0x80 (bit 7 set)

The handler filters out break codes by checking bit 7.

### How — Ring Buffer

The IRQ handler and the main loop run in different contexts. Doing translation and terminal output inside the IRQ handler is risky — it takes time and the handler could be re-entered or delay other interrupts. Instead, a **ring buffer** decouples the two:

- **IRQ handler** (`keyboard_handler`): reads the scancode from port 0x60, checks it is a make code, and enqueues it into `scancode_buf`. Returns immediately.
- **Main loop** (`keyboard_poll`): dequeues each raw scancode, looks it up in the translation table, and prints the result.

```
IRQ fires → keyboard_handler → scancode_buf (ring buffer)
                                       ↓
                              keyboard_poll (main loop)
                                       ↓
                              scancode_table lookup → terminal_putchar
```

The buffer is a 256-byte array with `uint8_t` head and tail indices. Because `uint8_t` overflows at 256 (equal to the buffer size), incrementing the indices wraps automatically without a modulo operation. The buffer is full when `head + 1 == tail`; in that case the scancode is silently dropped rather than corrupting the indices.

```c
// IRQ handler (interrupt context)
uint8_t next = buf_head + 1;   // wraps at 256 automatically
if (next != buf_tail) {
    scancode_buf[buf_head] = scancode;
    buf_head = next;
}

// Main loop
while (buf_tail != buf_head) {
    uint8_t sc = scancode_buf[buf_tail++];  // buf_tail wraps automatically
    char c = scancode_table[sc];
    if (c) terminal_putchar(c);
}
```

### How — Lookup Table

A 128-entry `const char` array maps each make code to its printable ASCII character. Keys with no printable representation (shift, ctrl, alt, function keys, caps lock) map to `0` and are skipped. Backspace and tab are also mapped to `0` because `terminal_putchar` does not handle them.

### How — Initialisation

```c
void keyboard_init(void) {
    irq_install_handler(1, keyboard_handler);
    pic_unmask_irq(1);
}
```

This registers the handler and enables IRQ1 on the PIC. No other IRQ lines are affected.

## Boot Sequence (Final)

```
_start (multiboot2.asm)
  └─ main (kernel.c)
       ├─ gdt_init()          — install kernel GDT
       ├─ terminal_init()     — set up VGA output
       ├─ idt_init()          — load IDT, register all ISR and IRQ stubs
       ├─ [ISR tests]         — fire int $0x0, $0x3, $0x6 to verify handlers
       ├─ pic_init()          — remap PIC, mask all IRQs
       ├─ keyboard_init()     — register IRQ1 handler, unmask IRQ1
       ├─ sti                 — enable hardware interrupts
       └─ hlt / poll loop     — idle; on each IRQ wake, drain scancode buffer
```
